## Observaciones

### 1. El rating `4.7` ya no existía

El commit `d905c14` (2026-04-21 22:00) ya había revertido `4.8→4.7` a `4.8` en los 8 idiomas. Al empezar este brief el rating real en código era ya `4.8` en todos lados. El foco del PR se movió enteramente a la **extracción** a `shared/businessProfile.ts`.

### 2. reviewCount era inconsistente — normalizado a 310

Antes del refactor el reviewCount tenía 3 valores distintos:
- `307` — en schemas JSON-LD estáticos de `client/src/utils/seo-config.ts`, `category-license-free.tsx`, `category-licensed.tsx`, `client/index.html`.
- `300` — en schema de `client/src/pages/about.tsx` (lines 614-615, `ratingCount/reviewCount`).
- `310` — en fallbacks runtime (`businessStatsCache.ts`, `business-stats.ts`, `gbpSync.ts`, `useBusinessStats.ts`, `HomePageSEO.tsx`, `gbp-server.ts`).

Normalizado todo a **310**, alineado con el runtime real. El mensaje del commit de Ivan (`37cf42a`, "docs(geo): setup Google Places API") confirma `Review count: 310 (¡no 307, 300 ni 300+ como estaba hardcoded!)` — validado contra la primera respuesta real de Places API.

### 3. `buildAggregateRating()` NO estaba hardcoded

Contrario a lo que sugería el brief (`server/seoInjector.ts:1472 — aggregateRating baseline 307 reviews + rating hardcoded`), la función `buildAggregateRating()` (lines 1473-1482) YA leía de `getCurrentStats()` (cache runtime de `businessStatsCache.ts`) antes de este PR. No hubo que tocarla.

### 4. Strings de descripción SEO multi-idioma NO se refactorizaron

Hay ~150 strings de descripción con `4.8★ Google` embebido inline en varios idiomas. Refactorizarlos requeriría:
- Template literals por idioma con interpolación: `${BUSINESS_RATING_STR}★ Google.`
- Esto convierte ~150 strings literales en ~150 concatenaciones, un reformat masivo que el brief explícitamente pide evitar.
- Recomiendo PR aparte dedicado SOLO a esto (y con tests regression SERP), si se quiere extracción total.

### 5. Los i18n keys (`googleRating`, `rating: '4.8 Google'`) NO se refactorizaron

Son strings localizados (formato y contexto varía por idioma: "4.8/5 en Google" vs "4.8 stars" vs "4.8 étoiles sur Google"). Meter una constante dentro de un sistema i18n rompe la abstracción de traducciones — cada idioma debe poder ajustar el formato.

### 6. Review count oficial Google = 310 (no 307)

El header del comment en `shared/businessProfile.ts` documenta esto:
```
// Rating / review count: when the values change in Google Maps.
// Manual update until S3, where gbpSync will auto-refresh from
// Google Business Profile API into the `business_stats` DB row.
```

Cuando S3 se complete, la tabla `business_stats` recibirá values frescos via gbpSync y el fallback en `businessProfile.ts` será solo contingencia.

### 7. Falta refactor en 2 fallbacks i18n de componentes

`client/src/components/TrustBadges.tsx:13` y `client/src/components/booking-flow/BookingTrustBanner.tsx:28` tienen:
```tsx
{t.trustEscalation?.googleRating || "4.8 en Google"}
```

El fallback string ES está hardcoded. Lo dejo fuera del scope porque:
- La string va en un contexto i18n y debería reescribirse como `t.trustEscalation?.googleRating || \`${BUSINESS_RATING} en Google\`` — pero eso mezcla constante numérica con i18n textual ("en Google" es español, no traducible).
- Por consistencia, si se quiere refactor aquí, debería ir junto con el PR de i18n strings (punto 5 arriba).
